1. Introduction et Contexte

1.1 Contexte

Dans de nombreux établissements d'enseignement supérieur en Afrique francophone, la gestion des emplois du temps, le suivi des séances de cours et le calcul des vacations des enseignants reposent encore sur des processus manuels et des documents papier.

Cette situation engendre de nombreux problèmes : erreurs de calcul, pertes de documents, difficultés de traçabilité, retards dans le paiement des enseignants vacataires, et absence d'une vision consolidée de l'activité pédagogique.

1.2 Problématique

Problèmes identifiés :
  • Gestion manuelle et peu fiable des emplois du temps
  • Absence de contrôle de présence des enseignants
  • Suivi difficile de l'avancement des programmes pédagogiques
  • Calcul manuel et source d'erreurs des vacations
  • Manque de traçabilité des activités pédagogiques

1.3 Objectifs du Projet

EduSchedule Pro est une application web full-stack qui couvre l'intégralité du cycle de vie d'une séance de cours, depuis sa planification jusqu'au paiement de l'enseignant.

Objectifs principaux :
  • Digitaliser la gestion des emplois du temps
  • Mettre en place un système de pointage électronique via QR-Code
  • Assurer le suivi pédagogique via un cahier de texte numérique
  • Automatiser le calcul et la validation des fiches de vacation
  • Fournir des tableaux de bord avec statistiques en temps réel

1.4 Présentation de la Solution

EduSchedule Pro s'articule autour de cinq modules interdépendants qui couvrent tout le cycle de vie d'une séance de cours :

1
Planification — L'admin crée l'emploi du temps et génère les QR Codes
↓
2
Pointage — L'enseignant scanne le QR Code pour valider sa présence
↓
3
Cahier de texte — Le délégué saisit le contenu pédagogique
↓
4
Signatures — Délégué et enseignant signent numériquement
↓
5
Vacation — Calcul automatique, validation et paiement

3.3 Cas d'Utilisation Détaillés

UC1 — Gérer l'emploi du temps (Administrateur)

Acteur principal : Administrateur
Précondition : Être connecté avec le rôle administrateur
Scénario nominal :
  1. L'administrateur accède à la page Emploi du temps
  2. Il crée un nouveau planning en sélectionnant classe et semaine
  3. Il ajoute les créneaux (matière, enseignant, salle, horaires)
  4. Le système vérifie automatiquement les conflits
  5. L'administrateur publie l'emploi du temps
  6. Le système génère les QR Codes pour chaque créneau

UC2 — Pointer la présence (Enseignant)

Acteur principal : Enseignant
Précondition : Avoir un créneau planifié avec QR Code valide
Scénario nominal :
  1. L'enseignant ouvre la page de pointage
  2. Il scanne le QR Code affiché en salle (ou saisit le token)
  3. Le système vérifie la validité du token et la fenêtre horaire
  4. L'heure de début réelle est enregistrée
  5. Un statut (à l'heure / retard) est attribué automatiquement

UC3 — Remplir le cahier de texte (Délégué)

Acteur principal : Délégué de classe
Précondition : La séance a été pointée par l'enseignant
Scénario nominal :
  1. Le délégué accède à la page Cahiers de texte
  2. Il crée un nouveau cahier pour le créneau en cours
  3. Il saisit le titre, les points abordés et les travaux demandés
  4. Il appose sa signature numérique sur le pad de signature
  5. L'enseignant signe à son tour et clôture la séance

4. Architecture Technique

4.1 Vue d'Ensemble

EduSchedule Pro suit une architecture client-serveur avec séparation stricte entre le frontend React (SPA) et le backend PHP (API REST). Une base de données MySQL assure la persistance des données.

Couche Présentation — Frontend React

Application React 18 (SPA) avec React Router pour la navigation, Context API pour la gestion d'état global (Auth, Theme), et Axios pour les appels API. Bootstrap 5 assure le responsive design.

Couche Communication — API REST

Échange de données au format JSON entre le frontend et le backend. Toutes les routes protégées nécessitent un header Authorization Bearer JWT.

Couche Logique Métier — Backend PHP

API REST en PHP 8.2 organisée selon le pattern MVC. Gestion de l'authentification JWT, génération des QR Codes, export PDF (TCPDF), et validation des données.

Couche Persistance — MySQL

Base de données MySQL 8 (MariaDB 10.4) avec 14 tables organisées autour du concept central de la séance de cours. Clés étrangères pour garantir l'intégrité référentielle.

4.3 Flux d'Authentification JWT

  1. L'utilisateur envoie email + mot de passe via POST /api/auth.php?action=login
  2. Le serveur vérifie les credentials avec password_verify()
  3. Si valide, le serveur génère un token JWT signé avec HMAC-SHA256
  4. Le token (valide 24h) est retourné au client
  5. Le client stocke le token dans localStorage
  6. Chaque requête suivante inclut le token dans le header Authorization
  7. Le middleware jwt_helper.php vérifie la signature et l'expiration

6. Description des Modules

6.1 Module 1 — Gestion de l'Emploi du Temps

Ce module permet à l'administrateur de créer et gérer les plannings hebdomadaires par classe. Il inclut la détection automatique des conflits (enseignant déjà occupé ou salle déjà prise).

Fonctionnalités implémentées :
  • Création de plannings hebdomadaires avec créneaux détaillés
  • Détection automatique des conflits enseignant/salle
  • Publication / dépublication des emplois du temps
  • Modification et suppression des créneaux
  • Génération des QR Codes uniques par créneau
  • Vue grille hebdomadaire (Lundi → Samedi)
  • Filtrage par classe

6.2 Module 2 — Pointage QR-Code

Ce module gère le début officiel d'une séance via un système de QR-Code sécurisé. Chaque QR Code est unique, chiffré avec un token HMAC, et expire après une fenêtre horaire définie.

Fonctionnalités implémentées :
  • Génération de QR Codes sécurisés (token unique 32 caractères hex)
  • Scan via caméra (html5-qrcode) ou saisie manuelle du token
  • Validation temporelle (fenêtre de ±15 minutes)
  • Détection automatique des retards (> 15 min)
  • QR Code à usage unique (invalidé après scan)
  • Log de tous les pointages (réussis et échoués)

6.3 Module 3 — Cahier de Texte Numérique

Le cahier de texte est le document central du suivi pédagogique. Il est rempli par le délégué et co-signé numériquement par le délégué et l'enseignant via un pad de signature HTML5.

Fonctionnalités implémentées :
  • Saisie du titre, contenu et travaux demandés
  • Signatures numériques sur canvas HTML5 (signature_pad)
  • Workflow : brouillon → signé délégué → clôturé
  • Verrouillage après clôture (plus de modification possible)
  • Export PDF de chaque fiche (TCPDF)
  • Visualisation des signatures enregistrées

6.4 Module 4 — Fiche de Vacation et Paiement

La fiche de vacation est générée automatiquement à partir des cahiers de texte clôturés. Elle constitue la pièce justificative comptable pour le paiement des enseignants vacataires.

Fonctionnalités implémentées :
  • Génération automatique du montant (durée × taux horaire)
  • Workflow de validation : généré → visé (surveillant) → approuvé (comptable)
  • Export PDF comptable avec tableau des séances
  • Filtrage par mois et année
  • Historique des validations

6.5 Module 5 — Tableau de Bord et Statistiques

Chaque rôle dispose d'un tableau de bord personnalisé avec des KPIs adaptés à ses responsabilités.

Fonctionnalités implémentées :
  • KPIs en temps réel (classes, enseignants, séances, cahiers, vacations)
  • Graphique BarChart — séances par jour de la semaine (Recharts)
  • Graphique PieChart — statut des fiches de vacation
  • Notifications automatiques selon le rôle (mise à jour toutes les 30s)
  • Mode sombre / clair (persisté dans localStorage)
  • Sidebar rétractable avec navigation par rôle

9. Difficultés Rencontrées et Solutions

DifficultéCauseSolution Apportée
MySQL ne démarrait pas dans XAMPP Base de données corrompue suite à un arrêt brutal Suppression du dossier corrompu dans mysql/data/ et réimportation du script SQL
Erreur CORS entre React (port 3001) et PHP React avait changé de port automatiquement Modification de cors.php pour accepter dynamiquement tous les origines localhost
Token JWT non transmis pour le téléchargement PDF La méthode window.open() ne transmet pas les headers HTTP Passage du token en paramètre GET et lecture côté PHP avec $_GET['token']
La bibliothèque signature_pad ne se chargeait pas Problème de chunk JavaScript lors du chargement dynamique Réinstallation via npm install signature_pad --save
Composer non installé sur Windows Outil non préinstallé avec XAMPP Téléchargement et installation de Composer depuis getcomposer.org
Extension zip manquante pour TCPDF Extension PHP zip désactivée dans php.ini Installation depuis source via Composer avec --prefer-source
LocalStorage non partagé entre HTML et React Domaines différents (localhost vs localhost:3000) Passage du token via paramètres URL avec page AuthCallback dans React
